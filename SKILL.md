# PPT Agent

## 概述

把 PPT 制作拆成一条单模型串行流程：

1. 理解任务
2. 形成主题与风格草案
3. 先从本地模板库中筛选候选母版
4. 基于模板生成 3 个风格候选页
5. 让用户确认方向
6. 整理内容并分页策划
7. 上屏文案精修
8. 统一视觉规范
9. 逐页生成 SVG
10. 渲染成图片做视觉复检
11. 修正后再进入下一页
12. 整套完成后做跨页深修和质感润色

默认产物：

1. `brief.md`
2. `style_draft.md`
3. `style_decision.md`
4. `concepts/option_a.svg`, `option_b.svg`, `option_c.svg`
5. `concepts/option_a.png`, `option_b.png`, `option_c.png`
6. `outline.md`
7. `screen_copy.md`
8. `design_spec.md`
9. `page_family_plan.md`
10. `slides/slide_01.svg`, `slide_02.svg` ...
11. `previews/slide_01.png`, `slide_02.png` ...
12. `template_choice.md`

如果用户明确要求 `.pptx`，再在 SVG 稳定后进入导入或转换步骤。

## 快速开始

1. 运行 `scripts/init_ppt_workspace.py` 初始化工作目录。
2. 先写 `brief.md` 和 `style_draft.md`，先把主题、受众、页数和风格边界说清。
3. 先查看 `templates/` 里的母版库和 `templates/TEMPLATE_LIBRARY.md`，挑出 3-6 个与当前主题最匹配的模板方向。
4. 基于同一页内容和母版结构生成 3 个候选样张到 `concepts/`，并渲染成 PNG 预览图。
5. 把 3 个方案给用户选，用户未确认风格前，不要开始整套制作。
6. 风格确认后，再写 `outline.md`、`page_family_plan.md`、`screen_copy.md`、`design_spec.md`。
7. `page_family_plan.md` 里先分配每页的页面角色、页型、密度和模板来源，不要直接开画。
8. 每次只处理一页，SVG 生成后先渲染图片并视觉复检，再进入下一页。
9. 整套页面生成完后，必须横向对比所有页，做一轮跨页深修，检查是否同质化、是否缺少页面功能差异、是否需要补充克制的渐变与质感层。

## 核心原则

### 1. 同一个模型串行推进

- 不要把任务理解成多智能体协作
- 不要把“Research Agent / Design Agent / Planner Agent”当成必须存在的系统角色
- 这些只是可以借鉴的提示词能力块，本 skill 的执行主体始终是当前这个模型

### 2. SVG 是核心产物

- 每页输出为独立 SVG 文件
- SVG 必须可以直接预览
- 默认画布使用 16:9
- 所有文本、图形、连接线、卡片、图标都优先在 SVG 内表达
- 不要依赖外链字体、外链图片或网页运行时样式
- 任何页在交付前都必须先被渲染成图片并重新看一遍画面效果

### 3. 先内容，后版式

- 先明确一页的核心洞察
- 再列出支撑该洞察的证据、数据、图示和辅助说明
- 先把原始材料改写成适合屏幕阅读的短文案
- 在决定最终版式前，先检查本地模板库里有没有合适的母版
- 最后决定是 Bento Grid、对比双栏、流程图、大图注释还是数据卡片

### 4. 先精修上屏文案，再画图

- 不要把网页、论文、漏洞披露或会议纪要里的原文直接搬进 SVG
- 每个文本块都必须先压缩成屏幕语言，再进入版式阶段
- 如果一个容器放不下文本，先改写文本，不要先缩字号
- 默认优先短标题、短句、短 bullet、短标注

### 5. 一页只讲一件事

- 每页只能有一个主结论
- 如果同一页存在两个平级主结论，拆页
- 如果证据不足，不要硬凑成一页

### 6. 先确认风格，再批量生成

- 在整套 PPT 开始前，先和用户确认主题内容与风格草案
- 候选页优先基于本地模板库派生，而不是每次从零胡乱起稿
- 必须至少提供 3 个明显不同的候选页，不允许只换颜色不换结构
- 用户未明确选定前，不要默认某个方案直接做整套
- 如果用户要求混搭，先写入 `style_decision.md`，再进入完整制作

### 6.5 模板库优先于空白画布

- skill 内置 `templates/` 目录，存放一组原创 SVG 母版
- 候选页阶段必须先查看模板清单，再决定是否从零生成
- 模板的作用是提供结构、节奏、模块语法和视觉骨架，不是要求机械照抄文案
- 模板默认继承结构，不强制继承原始配色
- 做不同主题的 PPT 时，应优先保留模板骨架，再按当前主题重配颜色系统
- 如果模板与当前主题不完全匹配，应在保留骨架的前提下改写模块
- 如果当前主题需要新的页面类型，可以新增模板，但不要跳过模板库直接忘记沉淀

### 6.8 样式多样性必须被显式配置

- 样式多样性不是“多找几个模板”就会自然出现
- 先定义风格令牌，再定义页型覆盖，再定义整套节奏
- 风格至少应明确：
  - `mood`
  - `color_scheme`
  - `typography`
  - `card_style`
  - `gradients`
  - `elevation`
  - `decoration`
  - `slide_type_overrides`
- deck 开始逐页生成前，必须先写 `page_family_plan.md`
- 相邻页面应至少在以下项目中变化 2 项：
  - 信息密度
  - 卡片语法
  - 背景处理
  - 主要强调方式
  - 图形方向
- 连续三页使用同一种页面语法，默认视为失败信号

### 7. 用渲染图做视觉复检，不只看 SVG 代码

- 视觉问题常常在代码里看不出来，但在渲染图里一眼就会暴露
- 生成 SVG 后，先渲染成 PNG 或缩略图，再用视觉方式检查
- 如果画面显得乱、挤、冲突、重心失衡，就回到 SVG 与文案层重做
- 没有经过渲染图复检的页面，不算完成

### 8. 整套统一，不等于每页长得一样

- 整套 deck 的配色、字体、线条语言和气质可以统一
- 但不同页面必须按功能使用不同的卡片语法
- 开场页、对比页、流程页、指标页、方法论页、总结页，不应只靠换文案区分
- 至少要在卡片类型、描边方式、强调区结构、流程节点样式或模块节奏上形成差异
- 如果所有页都在复用同一种深色圆角卡片，只是换位置或换颜色，这仍然不算完成

### 9. 渐变和小巧思要克制

- 渐变不是装饰性默认选项，而是重点强化工具
- 优先把渐变用于关键字句、关键结论、流程终点、总结区和少量标签
- 整套优先只定义 2-3 组统一渐变，不要每页换一组
- 背景光晕、细线、描边色差只能弱于主体，不能制造额外重心
- 如果加完渐变以后页面更花、更吵、更像海报噪音，就应立即减弱或删除

## 标准工作流

### Step 1. 写 `brief.md`

至少写清：

- 主题
- 受众
- 使用场景
- 页数预估
- 语言
- 风格关键词
- 风格禁区
- 成功标准

### Step 2. 写 `style_draft.md`

至少写清：

- 主题主张
- 受众与使用场景
- 想传达的语气
- 想要的视觉方向
- 明确不要什么风格
- 适合作为候选页的代表性内容

写完后先和用户确认草案，再开始做候选页。

### Step 3. 先筛选模板母版

执行：

1. 先阅读 `templates/TEMPLATE_LIBRARY.md`
2. 按当前主题、场景和用户偏好，从 `templates/` 中挑出 3-6 个最接近的模板
3. 将筛选结果写入 `template_choice.md`

至少写清：

- 哪些模板被考虑
- 每个模板适合当前任务的原因
- 哪些模板被排除以及原因
- 最终用于生成候选页的 3 个母版

### Step 4. 基于模板生成 3 个候选页

要求：

- 三个方案必须在结构、节奏、气质上明显不同
- 三个方案使用同一组事实和相近文案，方便比较
- 三个方案优先从本地模板库挑选 3 个不同母版改写而来
- 每个方案都生成：
  - `concepts/option_x.svg`
  - `concepts/option_x.png`
  - 一段简短说明，解释该方案适合什么场景

生成后必须先自己看 `option_x.png`，再展示给用户。

### Step 5. 用户确认风格

- 让用户从 3 个方案中选一个，或指定混搭方向
- 将结论写入 `style_decision.md`
- 在用户明确确认前，不要开始整套制作

### Step 6. 写 `outline.md`

按页列出：

- 页标题
- 核心洞察
- 支撑证据
- 推荐视觉形式

不要在这一步写具体 SVG 代码。

### Step 6.5. 写 `page_family_plan.md`

按整套节奏先分配：

- Slide 编号
- 页面角色
- 页面类型
- 信息密度
- 主要模板参考
- 主要强调方式

硬规则：

- 连续两页以内可以同族，但不要连续三页同族
- 8-15 页 deck 至少包含：
  - 1 页开场主张页
  - 1 页核心判断页
  - 1 页流程或系统图页
  - 1 页对比或矩阵页
  - 1 页风险或限制页
  - 1 页收束页
- 若 deck 目标是发布会感、汇报感或安全简报感，应通过页型切换来制造节奏，而不是只通过换色

### Step 7. 写 `screen_copy.md`

按页把所有会上屏的文字精修成短文案，至少包括：

- 标题
- 副标题
- 核心结论区块
- 每个卡片/节点/注释框的标题与正文
- 页脚来源与备注

强制规则：

- 标题要像页面标题，不要像文章标题
- 卡片正文优先 2 行内解决
- bullet 只保留动作、结果、影响、处置
- 删除背景铺垫、重复修饰和解释性废话
- 先改写，再布局；先删减，再缩放

### Step 8. 写 `design_spec.md`

至少定义：

- 画布尺寸
- 配色
- 字体策略
- 卡片、标题、标签、流程箭头、注释框的统一样式
- 哪些页面适合 Bento Grid，哪些不适合
- 关键容器的文本预算和最小字号
- 允许使用的背景装饰与明确禁用的视觉噪音
- 视觉重心、辅助重心和低优先级装饰的层级边界
- 页面功能族
  - 哪些页是开场主张页
  - 哪些页是对比页
  - 哪些页是流程页
  - 哪些页是指标或漏斗页
  - 哪些页是总结收束页
- 每类页面允许使用的卡片语法
  - 实心卡片
  - 描边卡片
  - 条纹卡片
  - 标签胶囊
  - 流程节点
  - 引言或结论收束块
- 统一渐变策略
  - 允许哪几组渐变
  - 渐变用于什么层级
  - 哪些元素禁止使用渐变
- 模板继承策略
  - 当前 deck 主要参考哪几个模板
  - 哪些页沿用模板骨架
  - 哪些页需要混搭或自行补充结构图
  - 哪些模板只继承结构，不继承原始颜色
- 风格令牌
  - `mood`
  - `color_scheme`
  - `typography`
  - `card_style`
  - `gradients`
  - `elevation`
  - `decoration`
  - `slide_type_overrides`
- 页面角色与页型覆盖
  - `cover`
  - `core_conclusion`
  - `users_scenarios`
  - `growth_flywheel`
  - `comparison`
  - `risks`
  - `summary`
  - `process`
  - `matrix`
  - `system_diagram`

### Step 9. 逐页生成 SVG

生成每页时，必须同时满足：

- 固定画布
- 明确层级
- 文本可读
- 留白合理
- 无装饰性噪音
- 不超出边界
- 文本严格来自 `screen_copy.md`
- 风格严格服从 `style_decision.md`
- 优先复用选中的模板骨架和结构图语法

如果内容很多，先删减信息，不要缩到不可读。

### Step 10. 渲染成图片做视觉复检

每生成一页 SVG，都必须：

1. 用 `scripts/render_svg_preview.sh` 通过 `sharp` 渲染成 PNG
2. 打开 PNG 预览图，而不是只盯着 SVG 代码
3. 用“第一次看到这页的人”的视角检查画面

重点检查：

1. 有没有块和块在打架
2. 有没有背景字、网格、渐变或装饰抢主体
3. 视觉重心是否过多，是否一眼不知道先看哪里
4. 卡片、标签、标题和大字是否互相争抢注意力
5. 底部流程或页脚是否被压得太挤
6. 是否存在“内容没冲突，但整体还是显乱”的情况
7. 是否存在任何重叠、压线、贴边或结构图遮挡文字的情况

如果预览图看起来乱，优先：

1. 删装饰
2. 降低背景存在感
3. 减少高饱和强调区
4. 拉开区块距离
5. 砍掉低优先级信息

### Step 11. 修正后再进入下一页

每一轮修正后，再重新渲染并检查：

1. 主结论是否清楚
2. 字号是否足够
3. 文本是否溢出
4. 元素是否过密
5. 图文是否相关
6. 是否存在无意义装饰
7. 有没有哪个文本块还像文章段落
8. 有没有任何卡片需要读 3 秒以上
9. 是否存在“能改短但还没改短”的句子

如果发现问题，先重写文本，再微调布局；如果画面层级有冲突，先改结构与装饰，再改细节。

### Step 12. 整套深修与质感润色

整套页面初版完成后，必须额外做一轮 deck-level refinement：

1. 把全部预览图横向对比，而不是只逐页看
2. 检查是否出现“所有页都是同一种卡片模板”的问题
3. 检查相邻页面在结构和节奏上是否过于重复
4. 按页面功能重分卡片语法，让开场、流程、指标、范式、总结页都有自己的表达方式
5. 只在关键字句、关键结论和终点模块上补充克制的渐变与小巧思
6. 如果某个渐变、光晕或细线让页面变花，立即删掉
7. 检查哪些页面仍然过于文字化，优先用手作结构图替代说明性长文案
8. 对照 `page_family_plan.md` 检查节奏是否真正落地，而不是回退成“同一模板换内容”
9. 参考 `references/style-diversity-system.md`，检查是否形成了“风格令牌 + 页型分配 + 质量门槛”三层系统

完成标准：

1. 整体一眼看得出属于同一套 deck
2. 不同页面又能明显看出不同的模块语法和叙事节奏
3. 重点区域有质感，但不会全页都在发光或抢戏

### Step 11. 导入或导出

如果用户明确要 PPT：

- 优先把 SVG 作为每页素材导入 PowerPoint
- 若需要进一步编辑，可在 PowerPoint 中转换为形状
- 如果当前任务要求更强可编辑性，再考虑调用其他技能或工具生成 `.pptx`

## SVG 生成规则

详细规则只在需要时读取：

- [references/svg-rules.md](references/svg-rules.md)

默认要求：

- 每页一个独立 `<svg>`
- 固定 `viewBox`
- 使用内联样式
- 尽量使用基础图形、路径、文本、分组
- 不使用脚本
- 不依赖浏览器交互

## 提示词使用规则

不要把整段长 prompt 原封不动地塞给模型。
按阶段拼接 [references/prompt-kernels.md](references/prompt-kernels.md) 中的规则块。

推荐顺序：

1. 内容提炼 kernel
2. 草案确认 kernel
3. 风格候选 kernel
4. 页面策划 kernel
5. 上屏文案精修 kernel
6. 视觉规范 kernel
7. SVG 输出 kernel
8. 渲染图视觉复检 kernel
9. Deck-Level Refinement kernel

## 交付纪律

- 默认交付真实文件，不只给方案
- 默认交付中间稿和 SVG
- 默认交付候选页与预览图，不跳过用户风格确认
- 如果用户没要求 `.pptx`，不要强行先转 PPT
- 如果用户只给了主题，没有资料，先补内容整理，不要直接画页
- 最终答复里必须说明：
  - 生成了哪些文件
  - 哪个方案被用户选中
  - 哪些页已经完成
  - 哪些假设是你代填的
