# Prompt Kernels

按阶段拼接，不要一次性全塞。

## 1. 内容提炼 Kernel

- 你现在在为 PPT 生成内容，不是在写文章
- 目标是提炼可展示的信息，而不是覆盖全部信息
- 每页必须围绕一个核心洞察
- 禁止虚构数据、事实、来源和结论
- 图片和图表应服务论证，而不是装饰
- 原始资料只能当素材，不能直接当上屏文案

## 2. 页面策划 Kernel

- 先写“这一页想让观众记住什么”
- 再写“哪些证据支撑这个结论”
- 再写“最合适的视觉表达形式”
- 如果内容过载，优先拆页，不要硬塞

## 3. 草案确认 Kernel

- 先把主题内容、受众、场景和风格边界整理成简洁草案
- 明确写出“想要什么”和“不要什么”
- 在用户确认前，不要直接开始整套 PPT
- 如果用户信息不足，先代填合理假设，并明确标出来等待确认

## 4. 风格候选 Kernel

- 在开始候选页前，先查看本地模板库和模板说明
- 优先从 3 个不同的模板母版派生候选页，而不是每次从空白画布起稿
- 模板复用优先复用结构和模块关系，不默认复用模板原始颜色
- 每个候选方案都要明确自己的风格族和页型身份
  - 例如：`tech + cover`、`business + core_conclusion`、`creative + chapter_break`
- 基于同一页代表性内容，生成 3 个明显不同的候选方案
- 差异必须体现在结构、留白、强调方式、背景处理和画面节奏上
- 不允许只换配色或小装饰冒充 3 个方案
- 每个方案用一句话说明适用场景和风险
- 候选页的目的不是完稿，而是帮助用户快速选定视觉方向

## 4.5 页型分配 Kernel

- 在开始整套生成前，先为每一页分配页面类型和密度
- 页面类型至少从以下集合中选择：
  - `cover`
  - `core_conclusion`
  - `users_scenarios`
  - `growth_flywheel`
  - `comparison`
  - `risks`
  - `summary`
  - `process`
  - `matrix`
  - `system_diagram`
  - `dashboard`
  - `chapter_break`
- 连续两页以内可同族，但不要连续三页同族
- 相邻页至少变化 2 项：
  - 卡片语法
  - 信息密度
  - 主强调方式
  - 背景处理
  - 图形方向
- 先输出 `page_family_plan.md` 式分配结果，再进入 SVG

## 5. 上屏文案精修 Kernel

- 你现在不是在写文章，而是在写一页幻灯片上的可读文案
- 每个文本块都必须压缩成屏幕语言
- 优先短标题、短句、短 bullet、短注释
- 删除背景铺垫，保留动作、结果、影响、处置
- 不允许把安全披露、研究报告或网页原文整句搬上页面
- 如果文本超过容器承载能力，先重写，再考虑布局
- 一个卡片正文默认不超过 2 行
- 一个 bullet 默认不超过 28 个汉字或等效信息量
- 如果一句话可以拆成两行短句，优先拆，不要堆成长句

输出时先给结构化上屏文案，不要直接生成 SVG。推荐结构：

```json
{
  "slide_title": "",
  "slide_subtitle": "",
  "hero_lines": ["", "", ""],
  "sections": [
    {
      "title": "",
      "bullets": ["", ""]
    }
  ],
  "cards": [
    {
      "title": "",
      "lines": ["", ""]
    }
  ],
  "source_line": "",
  "note_line": ""
}
```

## 6. 视觉规范 Kernel

- 为整套 deck 先定义统一的颜色、字号层级、容器、圆角、边框和连接线规则
- 不要每页都换一套风格
- 先定义风格令牌，再定义页型覆盖
- 风格令牌至少包括：
  - `mood`
  - `color_scheme`
  - `typography`
  - `card_style`
  - `gradients`
  - `elevation`
  - `decoration`
  - `slide_type_overrides`
- 版式稳定优先于花哨效果
- 若使用 Bento Grid，必须有明确的信息分块依据
- 为主要容器提前声明文本预算，不要到 SVG 阶段再临时塞字
- 写清哪些背景元素能用，哪些会制造视觉噪音
- 限制同时出现的强强调区数量，避免画面到处都在叫
- 同时定义页面功能族和对应卡片语法，不要整套只靠一种卡片模板硬撑
- 提前规定哪几种渐变允许使用，以及它们只能服务哪些层级
- 如果用户给了参考模板截图或网页，先抽象其中的模块语法，再做原创变体
- 允许学习时间线、图表拼接、环状信息图等结构，但不要逐像素抄第三方模板
- 若参考模板的颜色不适合当前主题，应重建配色，而不是机械保留原色
- 当前 deck 若以“大会感 / 发布会感 / 安全简报感”为目标，应通过页型覆盖来拉开差异，而不是只靠大背景字或高饱和块

## 7. SVG 生成 Kernel

- 请为当前这一张幻灯片生成独立、可直接渲染的 SVG 代码
- 使用固定 16:9 画布
- 文本必须可读，层级必须清晰
- 使用几何形状、容器、连接线和色彩编码表达逻辑关系
- 只能消费已经精修好的上屏文案，不要回头抓取原始资料长句
- 优先保留字号和留白，必要时删字，不要机械缩小
- 背景大字、装饰线、网格和发光效果只有在不抢主体时才允许保留
- 若当前页属于流程、指标、总结或对比页，应切换到对应的模块语法，不要复用上一页的同款卡片
- 若模板库中已有适合的骨架，请优先复用其结构并按当前主题改写
- 当页面信息过于单调时，优先增加手作结构图，而不是先找网图贴上去
- 渐变优先用于关键字句、关键结论和终点模块，不要整页泛滥
- 不要输出说明文字，只输出 SVG 代码

## 8. 渲染图视觉复检 Kernel

- 你现在看的不是代码，而是渲染后的页面图片
- 用第一次看到这页的观众视角判断是否显乱、显挤、显冲突
- 检查主体是否一眼可见，是否存在多个区域同时抢视线
- 检查背景字、网格、渐变、发光或大装饰是否压过正文
- 检查各区块是否互相争抢注意力，而不是形成清晰主次
- 检查底部流程、页脚和角标是否被压在边缘造成拥堵
- 检查渐变和小巧思是否真的提升质感，还是已经制造噪音
- 如果画面乱，先改结构与装饰强度，再改细节
- 还要检查这页和前后页是否过于相似，是否真的承担了自己的页型身份
- 如有问题，返回修正后的 SVG，并说明删掉或减弱了什么元素

## 9. SVG 读感复检 Kernel

- 检查文本是否超出容器
- 检查字号是否过小
- 检查留白是否失衡
- 检查是否有不必要的装饰
- 检查图文关系是否真的支撑核心洞察
- 检查是否还有文章句式、长段落或读起来费力的卡片
- 检查 bullet 是否还能继续压缩
- 如有问题，先重写文本，再返回修正后的 SVG，而不是只写问题列表

## 10. Deck-Level Refinement Kernel

- 你现在看的不是单页，而是整套 PPT 的预览序列
- 检查是否存在“每页都像同一模板”的问题
- 在不破坏统一风格的前提下，为不同页面分配不同卡片语法和强调方式
- 按以下维度给整套做横向复检：
  - 风格令牌是否统一
  - 页型覆盖是否清晰
  - 节奏是否有起伏
  - 是否连续多页过于相似
  - 是否存在某种卡片语法被滥用
- 只在关键页补充渐变、弱光晕、细描边和高光文字
- 如果渐变让页面更花，就删掉而不是继续叠加
- 如果某页综合效果低于 5/10，不做小修，直接换结构重做
- 输出时直接给修正后的 SVG 或设计调整，不要只给审美评价
