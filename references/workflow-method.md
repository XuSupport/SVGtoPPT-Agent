# 单模型工作流

## 定位

这是一个单模型串行工作流，不是多智能体系统。

你要做的是：

1. 用同一个模型逐步完成内容理解、草案确认、候选页生成、分页策划、上屏文案精修、视觉规范和 SVG 生成
2. 把复杂度拆进“阶段”和“文件”，而不是拆进多个 agent

## 执行顺序

### 1. 任务理解

- 明确主题、受众、场景、页数、语言、风格边界
- 把任务写成 `brief.md`

### 2. 草案确认

- 先把主题内容和风格方向写成 `style_draft.md`
- 先与用户确认草案，再生成候选页
- 没有用户确认，不进入整套制作

### 3. 候选页生成

- 候选页前先查本地 `templates/` 模板库
- 优先从现有母版中挑出 3 个结构差异明显的模板做改写
- 模板可以借结构，但颜色应根据当前主题重新设定
- 使用同一页代表性内容，做 3 个明显不同的候选页
- 方案差异要体现在结构、节奏、留白、强调方式和气质上
- 每个候选页都渲染成 PNG，再自己看一遍
- 将 SVG 与 PNG 放到 `concepts/`

### 4. 风格定稿

- 让用户选择一个方案，或明确指定混搭方向
- 把最终决定写入 `style_decision.md`
- 同时明确当前 deck 的风格族
  - `business`
  - `tech`
  - `creative`
  - `minimal`
  - 或按主题自定义 `security-report / product-launch / editorial-tech`

### 5. 内容整理

- 从用户输入、附件和本地资料抽取可展示的信息
- 只保留对页面有价值的内容
- 如果用户提供了参考网页或截图，同时抽取可迁移的模块语法

### 6. 分页策划

- 先决定每页讲什么
- 再决定每页用什么视觉形式
- 决定视觉形式时，优先复用模板库中的结构母版或手作结构图
- 若用户给了参考模板风格，可把它拆成“背景语法 + 模块语法 + 连接语法”三个层次再映射
- 将结果写入 `outline.md`

### 6.5 页型与节奏分配

- 在开始逐页画图前，先写 `page_family_plan.md`
- 给每页指定页面类型，而不是只写“这页放什么内容”
- 推荐页型：
  - `cover`
  - `core_conclusion`
  - `users_scenarios`
  - `growth_flywheel`
  - `comparison`
  - `risks`
  - `summary`
  - `process`
  - `matrix`
  - `system_diagram`
  - `dashboard`
  - `chapter_break`
- 连续两页以内可以同族，但不要连续三页同族
- 相邻页至少在以下项目中变化 2 项：
  - 信息密度
  - 卡片语法
  - 背景处理
  - 主强调方式
  - 图形方向
- 开始画图前，先确认整套至少有一页低密度开场、一页核心判断、一页流程或系统图、一页对比或矩阵、一页风险或限制、一页收束页

### 7. 上屏文案精修

- 把原始资料改写成适合投屏阅读的短文案
- 逐块控制字数，不要整段照搬来源材料
- 每个容器先过文本预算，再进入 SVG
- 将结果写入 `screen_copy.md`

### 8. 风格统一

- 定义整套 deck 的颜色、字体、卡片、标签、线条和图标规则
- 把风格写成令牌系统，而不是几句模糊审美词
- 最低应明确：
  - `mood`
  - `color_scheme`
  - `typography`
  - `card_style`
  - `gradients`
  - `elevation`
  - `decoration`
  - `slide_type_overrides`
- 明确哪些背景装饰可以用，哪些会制造噪音
- 写入 `design_spec.md`
- 同时定义页面功能族，不要让所有页都复用同一种卡片模板
- 提前约定哪些页适合条纹卡片、哪些页适合描边卡片、哪些页适合流程轨道或收束块
- 如果整体风格已经确定，变化应落在模块语法和节奏上，而不是推翻整套配色
- 即使复用模板，也不要被模板原色绑定；配色系统应服务当前主题、品牌和语气
- 如果 deck 目标是更强的大会感或品牌感，应在统一风格下使用不同页型覆盖，而不是整套只复制一个母版

### 9. 逐页 SVG 生成

- 每次只生成一页
- 一页生成完成就立即检查和修正
- SVG 只能消费已经精修过的短文案
- 版式必须服从 `style_decision.md`
- 页面结构优先服从选中的模板骨架
- 文件名统一为 `slides/slide_01.svg` 这类形式

### 10. 预览图视觉复检

- 把 SVG 渲染成 PNG 预览图
- 优先使用 `scripts/render_svg_preview.sh`，默认通过 `sharp` 渲染
- 直接看图片，不要只看代码
- 检查视觉重心是否清楚，背景是否抢戏，区块是否互相打架
- 如果画面显乱，优先删装饰、减强调、拉间距，再考虑局部修饰
- 检查渐变、光晕和色彩强调是否只出现在该强调的地方
- 如果渐变导致主体发糊、对比不足或整页显花，默认回退

### 11. 读感复检

- 检查文本块是否仍然像文章摘要
- 检查卡片、流程节点和注释框是否被长句拖垮
- 优先改写文本，再决定是否微调布局

### 12. 跨页深修

- 整套页初版完成后，必须把全部 PNG 预览图放在一起横向看
- 检查是否出现“开场页、方法页、指标页、总结页全都长一个样”的问题
- 同一套 deck 可以统一配色，但不应统一成单一模板
- 按页面功能重排卡片语法、强调区和流程元素，让每页在统一中有差异
- 关键页可以补少量渐变、小光晕和精细描边，但要服务主结论，不要为了炫技而加
- 如果某页纯文字过多，优先尝试加入手作结构图、矩阵、路径图、漏斗或职责分工图
- 横向检查 `page_family_plan.md` 是否被真实执行
- 如果整套缺少节奏换挡，优先改页型，不优先改颜色
- 单页复检可按以下维度打分：
  - Layout Balance
  - Readability
  - Typography
  - Information Density
  - Color Harmony
- 若某页综合效果低于 5/10，不要微补丁，直接换结构重做

### 12. 导出或导入

- 如果用户需要 PPT，再导入 PowerPoint
- 如果用户只需要 SVG 页面，不要额外增加转换步骤

## 方法论要点

- 一页一个核心洞察
- 信息是经过提炼的半成品，不是原材料
- 上屏文字要先精修，不能把原始材料直接塞进 SVG
- 用户先确认主题与风格，再批量生成
- 视觉是否和谐要以渲染图为准，不以代码整齐为准
- 图片、图示、结构图和流程线是内容的一部分
- Bento Grid 只是可选布局，不是默认答案
- 可读性和结构稳定优先于视觉炫技
- deck 级别的一致性不是模板复制，而是语法统一后的变化
- 渐变和小巧思应当用于“被记住的点”，而不是把所有重点都做成重点
- 模板库的价值是减少从零起稿的随机性，并沉淀已验证的结构母版
- 真正的样式多样性来自“风格令牌 + 页型分配 + 质量门槛”，而不是模板数量本身
